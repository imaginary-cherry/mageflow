**useAppStartup.ts:**

1. **Define AppPhase type:**
   ```typescript
   export type AppPhase =
     | 'loading-settings'    // reading tauri-plugin-store
     | 'onboarding'          // no settings found — show onboarding
     | 'starting-sidecar'    // sidecar spawning, waiting for port
     | 'connecting'          // polling /api/health
     | 'ready'               // main UI
     | 'startup-error';      // timeout or hard failure
   ```

2. **Define status messages** mapped to phases:
   - `loading-settings` -> "Loading settings..."
   - `starting-sidecar` -> "Starting backend..."
   - `connecting` -> "Connecting to services..."
   - `ready` -> "Ready"
   - `startup-error` -> error message string

3. **Hook implementation `useAppStartup()`:**
   - State: `phase`, `port`, `statusMessage`, `errorMessage`
   - On mount (useEffect):
     a. Set phase `loading-settings`
     b. Call `loadSettings()` — if null, set phase `onboarding` and return
     c. Set phase `starting-sidecar`, status "Starting backend..."
     d. Retry loop (up to 20 attempts, 500ms apart): call `invoke<number>('get_sidecar_port')`. If returns > 0, break. If all fail, set phase `startup-error` with "Backend failed to start"
     e. Set phase `connecting`, status "Connecting to services..."
     f. Poll `GET http://127.0.0.1:{port}/api/health` with 30-second deadline (500ms between attempts). On 200, set phase `ready`
     g. If deadline exceeded, set phase `startup-error` with "Backend did not respond within 30 seconds"

4. **`onOnboardingComplete` callback:**
   - Called when Onboarding form submits successfully
   - Calls `invoke('restart_sidecar')` to spawn sidecar with new credentials
   - Transitions to `starting-sidecar` phase and continues the startup flow (steps d-g above)
   - Extract the startup polling logic into a reusable async function `pollForReady(port)` used by both initial startup and post-onboarding

5. **`retrySidecar` callback:**
   - For crash recovery: calls `invoke<number>('restart_sidecar')`
   - Resets to `starting-sidecar` phase and re-runs polling

6. **Return:** `{ phase, port, statusMessage, errorMessage, onOnboardingComplete, retrySidecar }`

7. **Important:** Use `isTauriEnvironment()` check from settingsStore. When NOT in Tauri (browser dev mode), skip invoke calls and go directly to `ready` with port from `VITE_API_URL` or default 8000. This allows frontend dev without Tauri running.

**SplashScreen.tsx:**

1. **Props:** `{ statusMessage: string }`
2. **Layout:** Full-screen centered container with dark/branded background
3. **Content:**
   - App logo or app name "Mageflow Viewer" as large heading
   - Animated spinner (use a CSS spinner or lucide-react Loader2 icon with `animate-spin`)
   - Status message text below spinner (from `statusMessage` prop)
4. **Styling:** Tailwind classes, smooth appearance. Use `transition-opacity duration-500` on the wrapper for the fade-out transition (parent controls visibility via phase state).
