**useAppStartup.ts:**

1. **Define AppPhase type with 4 distinct startup phases per user decision:**
   ```typescript
   export type AppPhase =
     | 'loading-settings'      // reading tauri-plugin-store
     | 'onboarding'            // no settings found â€” show onboarding
     | 'starting-sidecar'      // sidecar spawning, waiting for port
     | 'connecting-hatchet'    // polling health endpoint for Hatchet connectivity
     | 'connecting-redis'      // polling health endpoint for Redis connectivity
     | 'ready'                 // main UI
     | 'startup-error';        // timeout or hard failure
   ```

2. **Define status messages mapped to phases (4 named steps per CONTEXT.md locked decision):**
   - `loading-settings` -> "Loading settings..."
   - `starting-sidecar` -> "Starting backend..."
   - `connecting-hatchet` -> "Connecting to Hatchet..."
   - `connecting-redis` -> "Connecting to Redis..."
   - `ready` -> "Ready"
   - `startup-error` -> error message string

3. **Tray status helper â€” call on EVERY phase transition:**
   ```typescript
   async function updateTrayStatus(phase: AppPhase) {
     if (!isTauriEnvironment()) return;
     const statusMap: Record<AppPhase, string> = {
       'loading-settings': 'starting',
       'onboarding': 'starting',
       'starting-sidecar': 'starting',
       'connecting-hatchet': 'starting',
       'connecting-redis': 'starting',
       'ready': 'connected',
       'startup-error': 'disconnected',
     };
     await invoke('set_tray_status', { status: statusMap[phase] });
   }
   ```
   This function MUST be called every time `setPhase()` is called. Create a wrapper:
   ```typescript
   const transitionTo = (newPhase: AppPhase) => {
     setPhase(newPhase);
     updateTrayStatus(newPhase);
   };
   ```

4. **Hook implementation `useAppStartup()`:**
   - State: `phase`, `port`, `statusMessage`, `errorMessage`
   - On mount (useEffect):
     a. Call `transitionTo('loading-settings')`
     b. Call `loadSettings()` â€” if null, call `transitionTo('onboarding')` and return
     c. Call `transitionTo('starting-sidecar')` (status: "Starting backend...")
     d. Retry loop (up to 20 attempts, 500ms apart): call `invoke<number>('get_sidecar_port')`. If returns > 0, break. If all fail, call `transitionTo('startup-error')` with "Backend failed to start"
     e. Call `transitionTo('connecting-hatchet')` (status: "Connecting to Hatchet...")
     f. Poll `GET http://127.0.0.1:{port}/api/health` â€” the health endpoint MUST return a JSON body that distinguishes service states, e.g. `{ status: "ok", hatchet: "connected", redis: "connected" }` or `{ status: "degraded", hatchet: "connected", redis: "error", detail: "..." }`. Poll until `hatchet` field is "connected" (or overall status is not an error). If Hatchet cannot connect within 15 seconds, call `transitionTo('startup-error')` with "Could not connect to Hatchet"
     g. Call `transitionTo('connecting-redis')` (status: "Connecting to Redis...")
     h. Continue polling `/api/health` until `redis` field is "connected". 15-second deadline. On timeout, call `transitionTo('startup-error')` with "Could not connect to Redis"
     i. Call `transitionTo('ready')` (status: "Ready")

   **Note on health endpoint:** The backend `/api/health` endpoint must be able to report Hatchet and Redis connectivity separately. If the current implementation returns a simple `{"status": "ok"}`, the executor should check and update the health endpoint in the Python server to return `{"status": "ok"|"degraded"|"error", "hatchet": "connected"|"error", "redis": "connected"|"error"}`. If modifying the backend is not feasible within this plan, use a sequential approach: first poll until `/api/health` returns 200 (Hatchet phase), then poll a second endpoint or re-poll with a query param. The key requirement is that the user sees distinct "Connecting to Hatchet..." and "Connecting to Redis..." messages.

5. **`onOnboardingComplete` callback:**
   - Called when Onboarding form submits successfully
   - Calls `invoke('restart_sidecar')` to spawn sidecar with new credentials
   - Transitions to `starting-sidecar` phase and continues the startup flow (steps d-i above)
   - Extract the startup polling logic into a reusable async function `pollForReady(port)` used by both initial startup and post-onboarding

6. **`retrySidecar` callback:**
   - For crash recovery: calls `invoke<number>('restart_sidecar')`
   - Calls `transitionTo('starting-sidecar')` and re-runs polling

7. **Return:** `{ phase, port, statusMessage, errorMessage, onOnboardingComplete, retrySidecar }`

8. **Important:** Use `isTauriEnvironment()` check from settingsStore. When NOT in Tauri (browser dev mode), skip invoke calls (including set_tray_status) and go directly to `ready` with port from `VITE_API_URL` or default 8000. This allows frontend dev without Tauri running.

   - Status message text below spinner (from `statusMessage` prop)
