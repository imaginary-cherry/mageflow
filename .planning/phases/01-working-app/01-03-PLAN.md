  <name>Task 1: Create settings store module with validation</name>
  <files>frontend/src/stores/settingsStore.ts</files>
  <action>
Create `frontend/src/stores/settingsStore.ts` with the following:

1. **Types:**
   ```typescript
   export interface AppSettings {
     hatchetApiKey: string;
     redisUrl: string;
   }

   export interface ValidationResult {
     valid: boolean;
     hatchetError?: string;
     redisError?: string;
   }
   ```

2. **`loadSettings(): Promise<AppSettings | null>`:**
   - Call `load('settings.json')` from `@tauri-apps/plugin-store`
   - Get `hatchetApiKey` and `redisUrl` from store
   - If both exist and are non-empty strings, return `{ hatchetApiKey, redisUrl }`
   - If either is missing/empty, return `null` (triggers onboarding)
   - Wrap in try/catch — if store fails to load (e.g., running in browser dev mode without Tauri), return `null`

3. **`validateCredentials(settings: AppSettings): Promise<ValidationResult>`:**
   - Per user decision: "Validate credentials before saving — try connecting to both services."
   - Approach: Spawn a temporary sidecar (or use the already-running one) and hit its health endpoint to check connectivity. However, during onboarding the sidecar is NOT yet running. Instead, use a lightweight validation approach:
     - **Option A (preferred):** Call `invoke('restart_sidecar')` to spawn the sidecar with the provided credentials, then poll `/api/health` on the returned port. The health endpoint should report per-service status (`{ hatchet: "connected"|"error", redis: "connected"|"error" }`). Parse the response to determine which service(s) failed.
     - **Option B (fallback if sidecar not yet available):** Temporarily save settings, call `invoke('restart_sidecar')`, poll health, and if validation fails, delete the saved settings. This is less clean but workable.
     - **Option C (simplest, if backend supports it):** Add a `/api/validate-credentials` POST endpoint to the Python backend that accepts `{ hatchet_api_key, redis_url }` and attempts to connect to both services without starting the full app. Returns `{ hatchet: "ok"|"error: ...", redis: "ok"|"error: ..." }`.
   - The executor should evaluate which option is feasible given the sidecar lifecycle. Option A is preferred because it reuses existing infrastructure.
   - Return `{ valid: true }` if both services connect, or `{ valid: false, hatchetError: "...", redisError: "..." }` with per-field error messages.
   - When NOT in Tauri environment (browser dev mode), skip validation and return `{ valid: true }`.

4. **`saveSettings(settings: AppSettings): Promise<void>`:**
   - Load store via `load('settings.json')`
   - `store.set('hatchetApiKey', settings.hatchetApiKey)`
   - `store.set('redisUrl', settings.redisUrl)`
   - `store.save()` — explicit save for durability (per research: auto-save only fires on clean exit)

5. **`hasSettings(): Promise<boolean>`:**
   - Convenience wrapper: calls `loadSettings()`, returns `result !== null`

6. **Environment detection:**
   - Add a helper `isTauriEnvironment(): boolean` that checks `window.__TAURI_INTERNALS__` existence
   - When NOT in Tauri (browser dev), use `localStorage` as fallback so the onboarding flow can be tested without Tauri running
   - Keep the fallback simple: `localStorage.getItem/setItem` with JSON serialization

Export all functions and types: `AppSettings`, `ValidationResult`, `loadSettings`, `saveSettings`, `hasSettings`, `validateCredentials`, `isTauriEnvironment`.

**Onboarding.tsx:**

Create a full-screen onboarding component with:

1. **Layout:** Centered card on a branded background. Use shadcn Card component. App name "Mageflow Viewer" as heading. Subtitle: "Connect to your infrastructure to get started."

2. **Form fields (using react-hook-form + zod):**
   - "Hatchet API Key" — text input, required, non-empty validation
   - "Redis URL" — text input, required, must start with `redis://` or `rediss://`, default placeholder `redis://localhost:6379`

3. **Validation flow (per user decision: "Validate credentials before saving"):**
   - Client-side: zod schema validates format before submission
   - Server-side: On submit, call `validateCredentials({ hatchetApiKey, redisUrl })` from settingsStore
   - Show a "Validating..." loading state on the submit button during validation
   - On validation failure: show inline error under the SPECIFIC field that failed, using the `hatchetError`/`redisError` from `ValidationResult` (per user decision: "inline error under the specific field that failed, e.g. 'Could not connect to Redis at this URL'")
   - On validation success: call `saveSettings()` then `onComplete()` callback

4. **Submit handler:**
   - Disable submit button during validation
   - Call `validateCredentials(settings)` first
   - If `valid === false`: set field-level errors using `form.setError('hatchetApiKey', { message: result.hatchetError })` and/or `form.setError('redisUrl', { message: result.redisError })`
   - If `valid === true`: call `saveSettings(settings)`, then call `onComplete()` callback prop

5. **Props:** `{ onComplete: () => void }`

**SettingsDialog.tsx:**

Create a reusable dialog for editing settings after onboarding:

1. **Uses shadcn Dialog component** with controlled open state via props

2. **Props:**

     open: boolean;

3. **On open:** Load current settings via `loadSettings()` and populate the form

4. **Same form fields and validation as Onboarding** (extract shared form logic into a `SettingsForm` sub-component or shared hook to avoid duplication)

   - Call `invoke('restart_sidecar')` from `@tauri-apps/api/core` to restart with new credentials
   - Show success toast via sonner
   - Close dialog

6. **Cancel button** closes without saving

Both components use Tailwind CSS for styling, consistent with the existing shadcn/ui design system.
