<task type="auto">
  <name>Task 1: Compile Python sidecar with Nuitka</name>
  <files>frontend/src-tauri/binaries/</files>
  <action>
1. **Verify Nuitka is installed:**
   ```bash
   pip install nuitka==4.0.4
   ```
   If 4.0.4 is not available, install latest 4.x.

2. **Get the target triple:**
   ```bash
   rustc --print host-tuple
   ```
   Store result (e.g., `aarch64-apple-darwin` or `x86_64-apple-darwin`).

3. **Run Nuitka compilation from the `libs/mage-voyance/` directory:**
   ```bash
   cd libs/mage-voyance
   python -m nuitka \
     --mode=onefile \
     --output-dir=../../frontend/src-tauri/binaries/ \
     --output-filename=mageflow-server \
     --include-package=uvicorn \
     --include-package=fastapi \
     --include-package=anyio \
     --include-package=starlette \
     --include-package=rapyer \
     --include-package=redis \
     --include-package=thirdmagic \
     --include-package=visualizer \
     --include-package=pydantic \
     --follow-imports \
     visualizer/__main__.py
   ```

   If `--mode=onefile` causes issues (extraction latency or antivirus), fall back to `--mode=standalone` and note the directory structure.

4. **Rename to target-triple format:**
   ```bash
   mv frontend/src-tauri/binaries/mageflow-server frontend/src-tauri/binaries/mageflow-server-$(rustc --print host-tuple)
   ```

5. **Make executable:**
   ```bash
   chmod +x frontend/src-tauri/binaries/mageflow-server-*
   ```

6. **Smoke test the binary:**
   ```bash
   ./frontend/src-tauri/binaries/mageflow-server-* --port 9876 --host 127.0.0.1 --redis-url redis://localhost:6379 &
   sleep 3
   curl -s http://127.0.0.1:9876/api/health
   kill %1
   ```
   Expected: `{"status":"ok"}` from health endpoint.

   If the binary fails to start (common issues):
   - Missing package: add `--include-package=<missing>` and recompile
   - gRPC issue: add `--include-package=grpc --include-package=google.protobuf --include-package-data=hatchet_sdk` and recompile
   - If health endpoint requires Redis and Redis is not running, that is OK — the binary starting and accepting connections is sufficient proof. The health endpoint may need to be made independent of Redis connection.

7. **Add the binary to .gitignore** (it should not be committed — it is platform-specific and large):
   ```
   frontend/src-tauri/binaries/mageflow-server-*
   !frontend/src-tauri/binaries/.gitkeep
   ```

8. **If compilation fails entirely** (Nuitka incompatibility), document the error and create a workaround: leave the binary slot empty and add a dev-mode fallback in lib.rs that spawns `python -m visualizer` directly instead of the sidecar binary. This fallback allows Phase 1 to complete for local development; Phase 2 CI will need to resolve the Nuitka issue.
